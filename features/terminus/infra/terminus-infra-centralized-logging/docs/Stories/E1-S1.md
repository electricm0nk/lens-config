# E1-S1 — Loki S3 Credentials: Vault KV + ExternalSecret

## Story

**As an** operator,  
**I want** Loki S3 credentials stored in Vault and materialized into the cluster via ESO,  
**so that** Loki can authenticate to the Synology-hosted MinIO without credentials in git.

## Context

Loki prod uses the Synology-hosted MinIO at `http://10.0.0.2:9000` as its S3-compatible object storage backend. MinIO is already running and accessible on the Synology NAS. No in-cluster MinIO deployment is needed.

This story only needs to: store the MinIO access credentials in Vault KV, create an ExternalSecret to materialize them as `loki-s3-creds` in the `monitoring` namespace, and verify the `loki` bucket exists on the Synology MinIO. The credentials are consumed by E1-S2 (Loki prod Helm values).

**Relevant architecture sections:**
- `architecture.md` §Phase 0 — Prerequisites
- `architecture.md` §Component Decisions §1 (Storage section)

## Pre-Implementation Checklist

- [ ] `monitoring` namespace exists (pre-existing — prometheus deployed there)
- [ ] Vault is accessible and KV secrets engine at `secret/` is enabled
- [ ] Check `kubectl get clustersecretstore` — note the ESO ClusterSecretStore name
- [ ] Verify Synology MinIO is reachable from a cluster pod: `curl http://10.0.0.2:9000/minio/health/live` → 200 OK
- [ ] Verify `loki` bucket exists in MinIO console at `http://10.0.0.2:9001` — create if not present

## Implementation Tasks

### Task 1: Create the `loki` Bucket on Synology MinIO (if not already present)

Open MinIO console at `http://10.0.0.2:9001`. Log in with MinIO root credentials. If the `loki` bucket does not already exist, create it:
- Bucket name: `loki`
- Versioning: off
- Object locking: off

Alternatively via `mc` CLI:
```bash
mc alias set synology http://10.0.0.2:9000 <ACCESS_KEY> <SECRET_KEY>
mc mb synology/loki
mc ls synology/loki   # confirm
```

### Task 2: Store Loki S3 Credentials in Vault

Use the MinIO access key/secret that Loki will authenticate with (create a dedicated MinIO service account in the MinIO console under Identity → Service Accounts, or use the root credentials for homelab simplicity):

```bash
vault kv put secret/terminus/loki/s3-creds \
  access_key_id=<MINIO_ACCESS_KEY> \
  secret_access_key=<MINIO_SECRET_KEY>
```

Verify:
```bash
vault kv get secret/terminus/loki/s3-creds
```

### Task 3: Create ExternalSecret for Loki S3 Credentials

Create `platforms/k3s/k8s/monitoring/loki-s3-creds-es.yaml`:

```yaml
apiVersion: external-secrets.io/v1beta1
kind: ExternalSecret
metadata:
  name: loki-s3-creds
  namespace: monitoring
spec:
  refreshInterval: 1h
  secretStoreRef:
    name: vault-backend        # adjust to match kubectl get clustersecretstore
    kind: ClusterSecretStore
  target:
    name: loki-s3-creds
    creationPolicy: Owner
  data:
    - secretKey: access_key_id
      remoteRef:
        key: secret/terminus/loki/s3-creds
        property: access_key_id
    - secretKey: secret_access_key
      remoteRef:
        key: secret/terminus/loki/s3-creds
        property: secret_access_key
```

Apply and verify:
```bash
kubectl apply -f platforms/k3s/k8s/monitoring/loki-s3-creds-es.yaml
kubectl get externalsecret loki-s3-creds -n monitoring   # should show READY=True
kubectl get secret loki-s3-creds -n monitoring           # should exist with both keys
```

### Task 4: Verify Cluster → Synology MinIO Connectivity

Confirm a pod in `monitoring` can reach the NAS S3 endpoint:

```bash
kubectl run -n monitoring s3-test --image=curlimages/curl --restart=Never --rm -it -- \
  curl -v http://10.0.0.2:9000/minio/health/live
# Expected: HTTP 200 OK
```

## Acceptance Criteria

- [ ] `loki` bucket exists on Synology MinIO (verified via MinIO console or `mc ls`)
- [ ] Loki S3 credentials stored in Vault KV at `secret/terminus/loki/s3-creds`
